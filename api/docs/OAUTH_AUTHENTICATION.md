# Google OAuth 2.0 Authentication - Documentation

## Overview

This project implements Google OAuth 2.0 authentication using the Authorization Code Flow directly in FastAPI, without Supabase Auth. Users authenticate with their Google account, and the server issues a JWT (JSON Web Token) for session management.

Supabase is used only as a PostgreSQL database — not for authentication.

---

## How the Authentication Flow Works

1. The user clicks "Login with Google" on the frontend.
2. The frontend redirects to `GET /auth/login`.
3. The server builds a Google authorization URL and redirects the user to Google's consent screen.
4. The user authorizes the app on Google.
5. Google redirects the user back to `GET /auth/callback` with a temporary `code` in the URL.
6. The server exchanges that `code` for an access token by making a POST request to Google's token endpoint.
7. The server uses the access token to fetch the user's info (email, name, profile picture) from Google's userinfo API.
8. The server creates or updates the user in the `users` table in PostgreSQL.
9. The server generates a JWT signed with `APP_JWT_SECRET` and returns it to the client.
10. The frontend stores the JWT and sends it in the `Authorization: Bearer <token>` header on every subsequent request.

## Environment Variables

All variables are required. See `.env.example` for reference.

| Variable | Purpose |
|---|---|
| `GOOGLE_CLIENT_ID` | Identifies the app to Google. Obtained from Google Cloud Console → APIs & Services → Credentials → OAuth 2.0 Client ID. |
| `GOOGLE_CLIENT_SECRET` | Private key to verify the app's identity. Same location as Client ID. Never share publicly. |
| `GOOGLE_REDIRECT_URI` | The URL Google redirects to after authorization. Must match exactly what is registered in Google Cloud Console. Development value: `http://localhost:8000/auth/callback` |
| `APP_JWT_SECRET` | Secret key used to sign and verify JWTs generated by the server. Generate with: `python3 -c "import secrets; print(secrets.token_hex(32))"` |
| `DATABASE_URL` | PostgreSQL connection string from Supabase. Found in Supabase Dashboard → Connect → Transaction pooler. |
| `CORS_ORIGINS` | Comma-separated list of allowed frontend origins. Development value: `http://localhost:5173` |

## API Endpoints

### `GET /auth/login`

Redirects the user to Google's consent screen. No parameters needed.

**Usage:** Open `http://localhost:8000/auth/login` in a browser or redirect from the frontend.

### `GET /auth/callback`

Receives the authorization code from Google, exchanges it for tokens, fetches user info, saves/updates the user in the database, and returns a JWT.

**Query parameter:** `code` (provided automatically by Google via redirect)

**Response:**
```json
{
  "token": "eyJhbGciOiJIUzI1NiIs...",
  "user": {
    "email": "user@gmail.com",
    "name": "User Name"
  }
}
```

### `GET /auth/me` (protected)

Returns the authenticated user's data from the JWT. Requires the `Authorization: Bearer <token>` header.

**Response:**
```json
{
  "google_id": "1023721194...",
  "email": "user@gmail.com",
  "exp": 1781988156
}
```

**Error responses:**
- `401 Unauthorized` — "Token expired" or "Invalid token"

## Protecting Routes

To protect any route, add `get_current_user` as a dependency:

```python
from fastapi import Depends
from app.dependencies.auth import get_current_user

@router.get("/some-protected-route")
async def protected_route(user=Depends(get_current_user)):
    # user contains: google_id, email, exp
    return {"message": f"Hello {user['email']}"}
```

## Database: Users Table

The `users` table stores authenticated users:

| Column | Type | Description |
|---|---|---|
| `id` | UUID (PK) | Auto-generated unique identifier |
| `google_id` | VARCHAR(255), UNIQUE | Google's unique user ID (never changes) |
| `email` | VARCHAR(255), UNIQUE | User's Google email |
| `name` | VARCHAR(255) | User's display name |
| `profile_picture` | TEXT | URL of the user's Google profile photo |
| `created_at` | TIMESTAMP WITH TIME ZONE | When the user first registered |
| `updated_at` | TIMESTAMP WITH TIME ZONE | When the user's data was last updated |

On repeated logins, the user's email, name, and profile picture are updated via `ON CONFLICT` (upsert).

## Google Cloud Console Setup

The following is already configured for this project:

1. **Project:** Created in Google Cloud Console.
2. **OAuth Consent Screen:** Configured with app name, external users, scopes: `email` and `profile`.
3. **OAuth 2.0 Client ID:** Type "Web application", with redirect URI `http://localhost:8000/auth/callback`.
4. **Google Calendar API:** Enabled (for future use).

If you need to access or modify these settings: [Google Cloud Console](https://console.cloud.google.com) → APIs & Services.

## Extending for Google Calendar (Future)

When the team is ready to implement Google Calendar integration, the following changes are needed:

1. **Add the Calendar scope to the login endpoint** in `app/routers/auth.py`:
   ```python
   "scope": "openid email profile https://www.googleapis.com/auth/calendar.readonly",
   ```

2. **Store the Google tokens** — add `google_access_token`, `google_refresh_token` columns to the `users` table and save them in the callback.

3. **Use the access token to call the Google Calendar API:**
   ```
   GET https://www.googleapis.com/calendar/v3/calendars/primary/events
   Authorization: Bearer <google_access_token>
   ```

4. **Handle token refresh** — when the access token expires (~1 hour), use the refresh token to get a new one by POSTing to `https://oauth2.googleapis.com/token` with `grant_type=refresh_token`.

## Running the Server

```bash
cd api
uv run uvicorn app.main:app --reload
```

## Testing the Flow

1. Open `http://localhost:8000/auth/login` in your browser.
2. Authorize with your Google account.
3. Copy the `token` from the JSON response.
4. Test a protected route:
   ```bash
   curl -H "Authorization: Bearer YOUR_TOKEN" http://localhost:8000/auth/me
   ```
