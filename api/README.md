# Pair Scheduling API

Backend for the Pair Scheduling platform — a CodeYourFuture tool where volunteers offer 1:1 slots and trainees book them.

Built with FastAPI, asyncpg (direct PostgreSQL, no ORM) and Supabase as the hosted database.

## Stack

- **Python 3.12+**
- **FastAPI** — web framework
- **asyncpg** — async PostgreSQL driver (no ORM)
- **Pydantic v2** — input/output validation
- **bcrypt** — password hashing
- **PyJWT** — JWT signing (HS256)
- **uv** — package manager and virtual environment

## Prerequisites

- Python 3.12 or higher
- `uv` installed locally

Install `uv` if you don't have it:

```bash
# macOS / Linux
curl -LsSf https://astral.sh/uv/install.sh | sh

# Or with Homebrew
brew install uv
```

## Setup

From the `api/` folder:

```bash
# 1. Install dependencies (creates the virtual env automatically)
uv sync

# 2. Copy the example env file and fill in the real values
cp .env.example .env
```

Then edit `.env` with your real values:

- `DATABASE_URL` — Supabase connection string. Get it from the Supabase dashboard: `Settings → Database → Connection String → URI`.
- `JWT_SECRET` — generate a long random string. Example: `openssl rand -hex 32`.
- `JWT_EXPIRES_MINUTES` — token lifetime in minutes (default `60`).
- `MIN_NOTICE_HOURS` — minimum hours of notice required to book a slot (default `24`).
- `CORS_ORIGINS` — allowed frontend origin. Default `http://localhost:5173` for local Vite dev.

## Run the dev server

```bash
uv run uvicorn app.main:app --reload
```

The API will be available at `http://localhost:8000`.

Interactive docs (auto-generated by FastAPI):

- Swagger UI: `http://localhost:8000/docs`
- ReDoc: `http://localhost:8000/redoc`

## Run the tests

```bash
uv run pytest
```

## Database migrations

This project does not use an ORM or a migration tool like Alembic.

Schema changes live as numbered `.sql` files in `migrations/`:

```
migrations/
├── 001_initial_schema.sql
├── 002_add_some_column.sql
└── ...
```

To apply a migration:

1. Open the Supabase dashboard.
2. Go to `SQL Editor`.
3. Paste the contents of the migration file and run it.
4. Apply migrations in numerical order, one at a time.

When you add a new schema change, create a new file with the next number — never edit an existing migration that has already been applied.

## Conventions

- All code, comments and commit messages in English.
- No comments in source code unless a non-obvious constraint must be explained.
- SQL is written directly with `asyncpg` — no ORM.
- Roles: `volunteer`, `trainee`, `admin`. Authorization enforced via FastAPI dependencies.
- Environment variables are loaded from `.env` — never hardcode secrets.
